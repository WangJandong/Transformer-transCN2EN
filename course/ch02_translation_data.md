# 第 2 章：认识翻译数据

## 本章目标

1. 理解平行语料的结构（一行中文 = 一行英文）
2. 知道 BPE 分词是什么，为什么中英翻译用共享词表
3. 会用 SentencePiece 对中文句子编码和解码

---

## 2.1 平行语料长什么样

机器翻译需要**平行语料**——每行源语言对应一行目标语言：

```
train.zh（中文）              train.en（英文）
────────────────────────────────────────────────────
今天天气很好。                 The weather is nice today.
我在学习机器翻译。             I am learning machine translation.
```

我们的项目有 **1884 万**行这样的句对，来自 WMT 中英翻译语料。

---

## 2.2 为什么要分词

模型不认识汉字，只认识数字。分词器把文本变成整数序列：

```
"今天天气很好" → [265, 108, 572, 9]
```

三种分词策略：

| 方案 | 示例 | 问题 |
|------|------|------|
| 按字 | 今/天/天/气/很/好 | 序列长，语义被拆散 |
| 按词 | 今天/天气/很好 | 中文词无限多，词表爆炸 |
| **BPE 子词** | ▁今天/天气/很好 | 高频词完整保留，低频词拆成片段 |

BPE（Byte Pair Encoding）是当前 NMT 的标准选择。

### 为什么不用结巴分词？

做中文 NLP 的同学通常会想到 [结巴分词](https://github.com/fxsjy/jieba)。两者核心区别：

| 维度 | BPE (本项目) | 结巴分词 |
|------|-------------|---------|
| **切分粒度** | 子词：`自然` `语言` `处理`（可拆可合） | 词：`自然语言` `处理`（以词典词为界） |
| **语言覆盖** | 中英共享一个词表 | 主要面向中文，英文另处理 |
| **OOV 处理** | 天然无 OOV，任何词可拆到字符 | 词典外的新词/人名可能切错或 OOV |
| **算法原理** | 统计相邻字符对频率，贪心合并 | 基于词典 + HMM/CRF 序列标注 |
| **词表大小** | 固定 32K，可控 | 词典大小无上限，动辄几十万 |
| **典型用途** | 神经翻译、大模型预训练 | 传统中文 NLP（分类、NER、搜索） |

**举个例子**：

```
句子: 自然语言处理正在改变世界
结巴: [自然语言] [处理] [正在] [改变] [世界]
BPE:  [自然] [语言] [处理] [正在] [改变] [世界]
```

结巴把"自然语言"当成一个词，BPE 把它拆成"自然"+"语言"——因为语料里"自然"和"语言"各自与其他词组合的频率更高，BPE 的贪心合并算法不会把它们锁死在一起。

**翻译任务为什么选 BPE 不选结巴**：

1. **中英粒度不对等**：结巴切中文词，但英文没有对应的"词"切分方式，两边 token 数差异大
2. **OOV 灾难**：翻译遇到人名、新词、拼写错误，结巴词典覆盖不了
3. **embedding 维度爆炸**：结巴词表可能几十万，BPE 固定 32K 可控
4. **迁移性**：BPE 中英共享词表，`▁DNA` 在两边是同一个 token id，模型天然对齐

---

## 2.3 BPE 怎么工作的

BPE 的核心思想：从字符开始，反复合并最频繁的相邻字符对。

### 举个简化例子

假设语料只有 3 句：
```
low
lower
newest
```

**第 0 步（初始）**：每个字符就是一个 token
```
[l][o][w]              → low
[l][o][w][e][r]        → lower
[n][e][w][e][s][t]     → newest
```

**第 1 步**：统计相邻字符对，发现 `e`+`s` 出现最频繁 → 合并为 `es`

**第 2 步**：`w`+`es` 最频繁 → 合并为 `wes`

**第 3 步**：`l`+`o` 出现 2 次 → 合并为 `lo`

**第 4 步**：`lo`+`w` 出现 2 次 → 合并为 `low`

最终词表： `[low, lower, newest, wes, lo, es, l, o, w, e, r, n, s, t]`

高频组合被"记住"成完整 token，罕见组合保持拆散状态。

### 在我们的项目里

- `自然` 经常共现 → 合并成一个 token
- `语言` 经常共现 → 合并成一个 token
- `饕餮` 太罕见 → 拆成 `饕` `餮`

英文同理：`Trans` + `form` + `er` = Transformer，因为完整词出现的频率不足以成为一个独立 token。

---

## 2.4 为什么中英共享词表

我们的分词器把中文和英文语料**合并在一起**训练。这样：

```
▁DNA → 同一个 token id
中文 "▁DNA" 和英文 "▁DNA" 用同一个向量表示
```

好处：
- 模型天然知道 "▁DNA" 在中英两边是同一个东西
- 源和目标 embedding 在同一个语义空间，利于对齐
- 参数量减半（不需要两个独立词表）

## 2.5 动手：编码一个句子

```python
import sentencepiece as spm

sp = spm.SentencePieceProcessor()
sp.load("spm_bpe.model")

# 编码：文本 → id 序列
text = "今天天气很好。"
ids = sp.encode(text, out_type=int)
print(ids)  # [265, 108, 572, 9]

# 解码：id 序列 → 文本
print(sp.decode(ids))  # 今天天气很好。

# 加特殊标记
bos_id, eos_id = 1, 2
model_input = [bos_id] + ids + [eos_id]
print(model_input)  # [1, 265, 108, 572, 9, 2]
```

特殊标记含义：
- `<bos>` (1)：句子开头
- `<eos>` (2)：句子结尾
- `<pad>` (0)：填充（不等长句对齐用）
- `<unk>` (3)：未知词

## 2.6 实际分词效果

用我们训练好的 `spm_bpe.model` 编码几个句子：

| 原文 | 分成的 pieces | ids |
|------|--------------|-----|
| `今天天气很好。` | `▁今天` `天` `气` `很` `好` `。` | `[1730, 28680, 28836, 28681, 28668, 28549]` |
| `我爱自然语言处理。` | `▁我爱` `自然` `语言` `处理` `。` | `[13761, 10652, 18447, 8091, 28549]` |
| `Transformer模型很强大。` | `▁Trans` `form` `er` `模型` `很` `强` `大` `。` | `[4988, 517, 14, 19973, 28681, 28946, 28593, 28549]` |
| `Hello world` | `▁Hell` `o` `▁world` | `[21098, 28527, 809]` |

注意 `▁` 代表**词的开头**（类似空格），是 SentencePiece 的标记。

可以看到：
- **高频中文词组**（`自然` `语言` `处理` `模型`）被保留为完整 token
- **低频单字**（`天` `气` `很` `好`）保持独立
- **英文词**被拆成常见子词片段（`Trans` `form` `er`），因为完整词出现频率不够
- 所有 piece 共享同一个 32K 词表空间

## 2.7 词表是怎么训练出来的

`spm_bpe.model` 不是"存词表"，而是存了一套 **merge 规则序列**。训练过程：

1. 把 `data/train.zh` + `data/train.en` 合并作为输入
2. SentencePiece 从字符级别开始，统计所有相邻字符对
3. 反复合并最频繁的那一对，直到凑够 32K 个 token
4. 输出 `.model`（merge 规则）和 `.vocab`（供人查看的词表）

训练代码见 [tokenizer.py](../tokenizer.py)：

```python
spm.SentencePieceTrainer.train(
    input=["data/train.zh", "data/train.en"],
    vocab_size=32000,
    model_type="bpe",
    character_coverage=0.9995,
    train_extremely_large_corpus=True,
    input_sentence_size=10_000_000,   # 从全量语料中采样 1000 万句训练
    shuffle_input_sentence=True,
)
```

### `input_sentence_size` 是什么意思？

我们的语料有约 1884 万对 = 3768 万行句子（zh + en）。
`input_sentence_size=10_000_000` 让 SentencePiece **只采样 1000 万句** 来学习 BPE merge 规则，目的是**加速 tokenizer 训练**。由于 `shuffle_input_sentence=True`，这 1000 万句是随机采样的。

对于中英翻译这种高频字词密集的语料，26% 的采样率已经足够学到高质量的 merge 规则。`character_coverage=0.9995` 也保证了字符覆盖度不会丢失。
